# План реализации Push Notifications & PWA (Вариант 1)

**Концепция:** Полный Self-Hosted (VAPID) с оптимизацией PWA для обхода ограничений iOS и умной маршрутизацией через PocketBase Hooks (ориентир - Telegram).

## Этап 1: Доработка Push Gateway (Node.js)
**Файл:** `infra/home/push-gateway/index.js`
* **Задача:** Реализовать детектирование протухших подписок.
* **Детали:** Web Push сервер (сервера Apple/Google/Mozilla) возвращает HTTP статус `410 Gone` или `404 Not Found`, если пользователь отписался, очистил кэш браузера или удалил PWA. Gateway должен перехватывать эти ошибки и возвращать в PocketBase отдельный массив `expired_endpoints: [url1, url2]`.

## Этап 2: Обработка ответов шлюза и очистка БД (PocketBase Hooks)
**Файл:** `infra/home/pb_hooks/task_helpers.js`
* **Задача:** Автоматическая очистка "мусорных" подписок.
* **Детали:**
  * В функции `handlePushTask` парсим JSON-ответ от Gateway (используя `$http.send().json()`).
  * Если в ответе есть `expired_endpoints`, через `$app.findRecordsByFilter` находим записи в коллекции `push_subscriptions` с этими `endpoint` и удаляем их (`$app.delete()`). Это предотвратит бесконечные попытки отправить пуши на мертвые устройства и сэкономит ресурсы.

## Этап 3: Умная маршрутизация и Presence (PocketBase Hooks)
**Файл:** `infra/home/pb_hooks/main.pb.js`
* **Задача:** Логика "Не спамить, если пользователь онлайн" и обогащение Payload.
* **Детали:** 
  * В хуке `onRecordAfterCreateSuccess` для сообщений получаем имя отправителя (через связь `sender.name`).
  * Получаем список `user_id` всех получателей комнаты.
  * Для каждого получателя делаем запрос к коллекции `presence`. Если `is_online == true` и `last_ping` обновлялся менее 60 секунд назад — мы считаем пользователя онлайн на десктопе или в открытом приложении. **Пуш не ставится в очередь** (или ставится с задержкой - опционально).
  * Формируем безопасный Payload: `{ title: senderName, body: "Новое сообщение", roomId: room.id, type: "NEW_MESSAGE" }`. Контент сообщения не передаем из соображений E2E безопасности.

## Этап 4: iOS PWA Prompt (Frontend)
**Файлы:** `app/src/components/ui/PwaInstallPrompt/` (новый)
* **Задача:** Обучение пользователей iOS для обхода ограничений Apple.
* **Детали:**
  * Apple (с iOS 16.4) разрешает Web Push **только** если сайт установлен на "Экран Домой".
  * Пишем компонент `PwaInstallPrompt`, который проверяет:
    1. Устройство iOS (`/iPad|iPhone|iPod/.test(navigator.userAgent)`).
    2. Приложение открыто в браузере Safari, а не как PWA (`!window.navigator.standalone`).
  * Если оба условия выполнены, показываем элегантную шторку снизу (BottomSheet): *"Для работы уведомлений добавьте приложение на экран Домой: нажмите значок «Поделиться» и выберите «На экран "Домой"»"*. 
  * Запоминаем в `localStorage` (например, `pwa_prompt_dismissed`), если пользователь закрыл подсказку, чтобы не бесить его.

## Этап 5: Навигация из уведомлений (Service Worker)
**Файл:** `app/src/sw.ts`
* **Задача:** Правильная обработка `notificationclick`.
* **Детали:**
  * В `sw.ts` в событии `notificationclick` извлекаем `roomId` из `event.notification.data.roomId`.
  * При открытии окна направляем пользователя не на `ROUTES.HOME`, а на `/chat/${roomId}`. 
  * Дорабатываем логику `client.focus()`: если приложение уже открыто, но на другом экране, нужно передать сообщение (postMessage) клиенту для программной навигации `router.navigate()`, чтобы не перезагружать SPA.

## Этап 6: Тестирование End-to-End
* Запустить `docker-compose` с Push Gateway.
* Отправить сообщение с одного браузера на другой (свернутый).
* Проверить доставку пуша, клик по нему (переход в нужный чат) и удаление подписки при симуляции ошибки 410.
