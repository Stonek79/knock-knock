# План доработки: Рассылка медиа-сущностей и управление (Broadcast)

## Обновленные требования
1. **Медиа-вложения**: Поддержка файлов, изображений, видео, аудио.
2. **Кэширование и рендер**: Учет работы браузерного HTTP-кэша и локального хранилища (Dexie) для публичных медиа, обход механизмов E2E-дешифровки.
3. **Отзыв рассылки**: Администратор должен иметь возможность удалить отправленную рассылку.
4. **Удаление без следа**: При удалении рассылки админом, сообщения должны исчезать у всех пользователей полностью (без "tombstone" плашек "Сообщение удалено").

---

## Архитектура и Решения

### 1. Работа с медиа и кэшем (Frontend)
Так как системные рассылки идут всем пользователям, использовать ключи комнат (`roomKey`) невозможно.
* **Загрузка**: Будет создан метод `uploadSystemMedia` в `media.repository.ts`, загружающий файлы в коллекцию `media` с флагом `is_vault=false` (без E2E).
* **Рендер**: В хуках или компонентах, рендерящих вложения (`MessageAttachments`), добавится проверка: `if (message.type === 'system')`. В этом случае медиа отображается напрямую по URL (`pb.files.getURL(...)`).
* **Кэш**: Публичные URL (`getURL`) отлично кэшируются самим браузером (ServiceWorker / HTTP Cache-Control). Нам не нужно прогонять эти файлы через `mediaDb` (Dexie) и процесс дешифровки (Web Crypto API). Это радикально снизит нагрузку на клиент при получении тяжелых системных сообщений.

### 2. Управление рассылками и функция отзыва
Чтобы иметь возможность удалить все копии системного сообщения у всех пользователей:
* **Связывание**: Эндпоинт генерации рассылки (`/api/custom/broadcast`) при создании задачи генерирует уникальный `broadcast_id`. Крон-воркер добавляет этот `broadcast_id` в `metadata` каждого сгенерированного сообщения.
* **Сохранение в БД**: Будет создана запись в специальной таблице или использована сама очередь задач (Task) как источник "Истории рассылок", чтобы администратор мог видеть список своих отправленных рассылок в UI админки.
* **UI Админки**: В `BroadcastSettings` под формой отправки добавится блок "Недавние рассылки" с кнопкой корзины.
* **Эндпоинт удаления (`DELETE /api/custom/broadcast/:id`)**:
  * Ищет все `messages`, где `metadata.broadcast_id = id` и `type = 'system'`.
  * Выполняет **жесткое удаление** (Hard Delete) этих сообщений через серверный API. Это гарантирует, что сообщения исчезнут без следа (так как мы удаляем запись из БД, а клиенты по реалтайму получат событие `DELETE` и уберут их из UI без замены на "Сообщение удалено").
  * Также удаляет связанные медиа-вложения (картинки, файлы) из БД PocketBase.

## Подробный план работ по шагам

### Шаг 1: Интеграция MessageInput и загрузка файлов
* В `media.repository.ts`: реализовать `uploadSystemMedia(formData)`.
* В `BroadcastSettings`: заменить `TextArea` на `MessageInput`.
* В `broadcast.ts`: прописать логику подготовки `attachmentIds` перед вызовом API рассылки.

### Шаг 2: Серверная логика рассылки
* Обновить `infra/home/pb_hooks/main.pb.js`: эндпоинт `POST /api/custom/broadcast` принимает `attachments`.
* В задаче для рассылки (`task`) генерируется уникальный идентификатор.
* В крон-обработчике: проставлять `attachments` и `metadata: { broadcast_id }` всем копиям сообщений.

### Шаг 3: Адаптация рендера чата
* Обновить компоненты отображения сообщений (`MessageBubble`, `AttachmentList`): для `message.type === "system"` отображать вложения по прямому URL, в обход `mediaDb` и дешифрации.

### Шаг 4: Реализация отзыва рассылки (Удаление)
* Добавить эндпоинт `DELETE /api/custom/broadcast/:broadcast_id` в `main.pb.js`, который делает жесткое удаление (`$app.delete()`) для всех сообщений рассылки и вложений.
* В `BroadcastSettings` создать компонент `BroadcastHistory` (список отправленных рассылок из таблицы задач/рассылок) с кнопкой "Удалить".
* Убедиться, что фронтенд при получении реалтайм-события `DELETE` полностью убирает `system` сообщение из списка чата, не оставляя плашку "Удалено".
