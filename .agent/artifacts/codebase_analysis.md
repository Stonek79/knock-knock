# Анализ кодовой базы: Особенности разработки JS-хуков PocketBase (v0.23+)

## 1. Описание инцидента с блокировкой транзакций SQLite
**Симптомы сбоя:**
При выполнении фоновых кронов (task_runner) или HTTP-запросов возникала критическая ошибка базы данных:
`sql: transaction has already been committed or rolled back`
После этого PocketBase переставал записывать логи, обрабатывать очередь задач и отсылать пуши, переходя в аварийный режим.

### Корневая причина (Root Cause)
В файле `infra/home/pb_hooks/security.pb.js` были объявлены обработчики событий создания, обновления и удаления записей (например, `onRecordCreateRequest`). Внутри этих обработчиков в некоторых ветках выполнения (особенно при валидациях) использовался обычный ранний возврат:
```javascript
if (isAdmin) {
    return; // ОШИБКА: транзакция зависает!
}
```
В PocketBase JSVM хуки событий выполняются в цепочке (middleware). Если обработчик не вызывает `e.next()`, цепочка прерывается. Поскольку системный хук создания/обновления записи оборачивается PocketBase в транзакцию базы данных SQLite, отсутствие вызова `e.next()` приводило к тому, что транзакция SQLite не завершалась (не делался ни `COMMIT`, ни `ROLLBACK`). Соединение оставалось заблокированным, «отравляя» пул соединений SQLite. При попытке запустить другие транзакции (например, в кронах воркера очереди) SQLite возвращал ошибку о том, что транзакция уже завершена или откачена.

## 2. Критический чек-лист при написании JS-хуков для PocketBase

### Правило 1: Обязательный вызов `e.next()` во всех ветках выполнения
Каждая ветка кода в хуках PocketBase (включая все блоки `if`, проверки условий и ранние возвраты) **обязана** вернуть или вызвать `e.next()`. Голые `return;` или `return false;` без вызова цепочки приводят к утечкам транзакций.
* **Неправильно:**
  ```javascript
  onRecordCreateRequest((e) => {
      if (e.record.get("status") === "draft") {
          return; 
      }
      // ... логика
      e.next();
  }, "posts");
  ```
* **Правильно:**
  ```javascript
  onRecordCreateRequest((e) => {
      if (e.record.get("status") === "draft") {
          return e.next(); 
      }
      // ... логика
      return e.next();
  }, "posts");
  ```

### Правило 2: Правильный жизненный цикл инициализации сервера
* Событие инициализации бэкенда называется `onBootstrap`, а **НЕ** `onAfterBootstrap`. Использование несуществующего события вызывает `ReferenceError` и полностью блокирует парсинг файла хуков сервером.
* Внутри `onBootstrap` база данных PocketBase еще не инициализирована до тех пор, пока не будет вызван `e.next()`. Если вы хотите сделать SQL-запросы к БД при старте сервера, вызовите `e.next()` **до** обращения к приложению.
* **Правильно:**
  ```javascript
  onBootstrap((e) => {
      e.next(); // Инициализируем БД
      
      // Теперь безопасно делать запросы
      const records = e.app.findAllRecords("users"); 
  });
  ```

### Правило 3: Транзакции и безопасность вложенных транзакций
Хуки событий (такие как `onRecordAfterCreateSuccess` или `onRecordCreateRequest`) по умолчанию уже выполняются внутри родительской SQLite-транзакции. 
* Использование вложенных транзакций через `e.app.runInTransaction()` внутри этих хуков крайне опасно и может нарушить целостность пула соединений SQLite (из-за ограничений вложенных SAVEPOINT).
* Вместо вложенных транзакций для безопасного сохранения записей в хуках следует использовать метод `e.app.saveNoValidate(record)` напрямую.

### Правило 4: Изоляция контекста выполнения JSVM
Каждый файл хуков (`*.pb.js`) и каждый обработчик выполняются в изолированных контекстах виртуальной машины JSVM.
* Глобальные переменные, объявленные в одном файле хуков, недоступны в другом.
* Для переиспользования общего кода используйте системную функцию `require()`.
* Глобальный объект приложения доступен как `$app` (в кронах и глобальных событиях), но внутри событийных хуков следует использовать контекстный объект события `e.app` (который привязан к текущей транзакции запроса).

## 3. Внедренная система самолечения (Self-Healing)

Для автоматического предотвращения и ликвидации зависаний очереди при сбоях были внедрены два механизма:

1. **Бутстрап-сброс задач (`onBootstrap`):**
   При каждом перезапуске PocketBase хук инициализации находит все задачи в коллекции `task_queue`, зависшие в статусе `processing`, и принудительно переводит их обратно в `pending`, сбрасывая блокировки.
2. **Воркер-таймаут (`task_runner` cron):**
   Каждую минуту крон-воркер проверяет выполняющиеся задачи. Если задача находится в статусе `processing` более 10 минут, она принудительно помечается как `failed` с сохранением ошибки `Task timed out in processing state` и инкрементом попыток (`attempts`). Это позволяет очереди не блокироваться из-за одного зависшего HTTP-запроса или транзакции.
