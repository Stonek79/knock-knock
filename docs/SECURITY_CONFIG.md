## 📁 Какие файлы можно коммитить в git

### ✅ Можно коммитить (безопасно)

| Файл | Причина |
|------|---------|
| `.env.example` | Шаблон без реальных значений |
| `*.env.example` | Любой шаблон окружения |
| `playwright.config.ts` | Конфигурация тестов без секретов |
| `scripts/use-env.sh` | Скрипт не содержит секретов |
| Документация (`.md`) | Если не содержит реальных ключей |

### ❌ НЕЛЬЗЯ коммитить (конфиденциально)

| Файл | Причина |
|------|---------|
| `*.env*` (кроме `.env.example`) | **Все .env файлы игнорируются через .gitignore** |
| `.env` | Содержит активные ключи доступа |
| `.env.test` | Содержит ключи тестового окружения |
| `.env.production` | Содержит production ключи |
| `.env.local` | Локальные секреты разработки |
| `*.wireguard_keys` | Приватные ключи VPN |
| `infra/home/pb.env` | Конфигурация PocketBase с паролями |
| `infra/scripts/.env` | Скрипты с локальными ключами |

---

### 🔒 .gitignore правило для .env файлов

В корне проекта настроено строгое правило:
- `*.env*` — игнорирует **все** файлы, содержащие `.env`.
- `!*.env.example` — исключение для шаблонов.

## 🔐 Какие данные являются чувствительными

### Критические секреты (никогда не коммитить!)

1. **PocketBase Keys:**
   - `PB_ENCRYPTION_KEY` — ключ шифрования данных в БД (очень важно!).
   - `ADMIN_EMAIL` / `ADMIN_PASSWORD` — доступ к панели управления.
   - `VITE_PB_URL` — публичный адрес API.

2. **Сетевые данные:**
   - Внутренние IP-адреса (`10.0.0.X`).
   - Конфигурация WireGuard (приватные ключи).
   - Порт контейнера: `8090` (Prod) / `9090` (Dev).

3. **E2EE Ключи:**
   - Ключи пользователей генерируются на клиенте и хранятся в `IndexedDB` / `Keystore`.
   - Никогда не логируйте приватные `x25519` записи.

## 🛡 Лучшие практики безопасности

### 1. Разделение сред

**Правильно:**
- `.env.test` → `knok-knok-pb-dev` (Dev-API).
- `.env.production` → `knok-knok-pb` (Prod-API).

### 2. PocketBase API Rules
Безопасность данных в PB строится на **API Rules**. 
- Всегда используйте фильтр `@request.auth.id != ""` для приватных данных.
- Для сообщений доступ должен быть разрешен только участникам комнаты.

### 3. Доступ через VPN (WireGuard)
PocketBase рекомендуется запускать внутри доверенной сети. VPS проксирует только HTTPS (8443) в туннель к домашнему серверу. Прямой доступ к порту 8090/9090 из интернета должен быть закрыт фаерволом (UFW).

### 4. Защита от ботов (Bot Protection)
В систему регистрации и входа внедрены механизмы пассивной защиты, не требующие ввода капчи:
- **Honeypot (`username_bot`)**: Скрытое поле в формах. Если оно заполнено — запрос автоматически отклоняется серверным хуком.
- **Time-check**: Сервер замеряет время между монтированием формы и её отправкой. Если прошло менее **3 секунд**, запрос считается автоматизированным и отклоняется.
- **Rate Limiting**: Ограничение количества попыток входа с одного IP (реализовано на клиенте через `useRateLimiter` и на сервере через встроенные механизмы PocketBase).

### 5. Безопасность фоновых задач (Task Runner)

Фоновая обработка (рассылка пушей, очистка данных) вынесена в изолированный Task Runner:
1. **Admin Only Access**: Коллекция `task_queue` имеет API Rule `viewRule: null`, что делает её невидимой и недоступной извне через любые клиентские запросы. Доступ к ней есть только у системных хуков и администраторов.
2. **Изоляция полезной нагрузки**: Все чувствительные данные (токены подписок, метаданные) в закрытой коллекции очереди, что предотвращает утечку при компрометации клиентского токена пользователя.
3. **Безопасные ретраи**: Ошибки операций записываются в поле `last_error`, но не выводятся пользователю, что исключает раскрытие внутренней структуры инфраструктуры.


## 💾 Пользовательские данные и хранение (Data Retention)

В Knock-Knock реализована политика автоматической очистки данных (Data Retention Policy) для минимизации цифрового следа и нагрузки на сервер:

1. **Media TTL (30 дней)**: Все зашифрованные файлы (изображения, видео, аудио) в коллекции `media` хранятся на сервере ровно **30 дней** с момента отправки.
2. **Task Runner**: Удаление производится автоматически фоновым процессом `pb_hooks/tasks.pb.js`.
3. **Local Store (Dexie)**: На стороне клиента (браузера) файлы также кешируются в IndexedDB на 30 дней.
4. **Исключение (Favorites)**: Медиафайлы в комнате «Избранное» (Self-Chat) **не подпадают** под автоматическое удаление и хранятся неограниченно (до ручного удаления пользователем).

## 🚨 Что делать при утечке секретов

### 1. Если скомпрометированы ключи PocketBase

```bash
# 1. Зайти в PocketBase Admin UI → Settings → Application
# 2. Обновить PB_ENCRYPTION_KEY и пересоздать Superuser
# 3. Перезапустить контейнер
docker compose -f docker-compose.pb.yml restart pocketbase

# 4. Задокументировать инцидент
```

### 2. Если утекли ключи WireGuard

```bash
# 1. Сгенерируйте новые пары ключей
wg genkey | tee privatekey | wg pubkey > publickey

# 2. Обновите конфиги на всех устройствах
# 3. Переподключите VPN
```

### 3. Если утекли SMTP-ключи (Brevo)

```bash
# 1. Зайти в панель Brevo → API Keys → Revoke
# 2. Сгенерировать новый ключ
# 3. Обновить .env файлы и перезапустить контейнер
```

## 🔍 Автоматическая проверка секретов

В проекте настроен **Husky pre-commit хук**, который автоматически проверяет индексированные файлы на наличие паттернов секретов (JWT, пароли, ключи WireGuard) перед каждым коммитом.

**Как это работает:**
1. Вы запускаете `git commit`.
2. Хук выполняет `git diff --cached` и ищет подозрительные строки.
3. Если секреты найдены, коммит блокируется.

**Файл хука:** `.husky/pre-commit`

## 7. Административные и технические скрипты

В папках `scripts/` и `infra/scripts/` содержатся мощные инструменты для управления базой данных (очистка, создание админов, посев данных). 

### Стратегии безопасности при хранении в Git

1. **Нет секретов в коде**: Все скрипты полагаются исключительно на переменные окружения (`.env`), которые исключены из Git.
2. **Защитные проверки (Guards)**:
   - Скрипты очистки данных требуют флаг `--force` при работе с удаленными базами.
   - Скрипты содержат обязательную задержку (5 секунд) перед деструктивными действиями.
   - Интерактивное подтверждение в Bash-обертках.
3. **Разграничение сред**:
   - Используйте `use-env.sh production` только в исключительных случаях.
   - Для обычной разработки и CI всегда используйте `staging` или `mock`.

### Рекомендации по использованию

- **GitHub Secrets**: При использовании скриптов в GitHub Actions передавайте `PB_ENCRYPTION_KEY` и `ADMIN_PASSWORD` через секреты репозитория, а не через файлы.
- **Admin Access**: PocketBase Superuser имеет полный доступ к данным. Ключи хранить как «Root-пароль» системы — только в защищённом менеджере паролей.
- **Аудит**: Любое использование административных скриптов на продакшене должно сопровождаться проверкой текущего контекста (правильный ли `.env` загружен).
