# План реализации: Современная медиа-система Knock-Knock (v3)

Этот документ описывает архитектуру и шаги внедрения системы обработки, шифрования и хранения медиафайлов, соответствующей принципам **Zero Knowledge**, **Offline First** и **Mobile First**.

## 🏗 Архитектурные принципы
- **Zero Knowledge**: Сервер хранит только зашифрованные данные и не имеет доступа к метаданным (именам, типам).
- **Offline First**: Использование IndexedDB (Dexie.js) для надежного локального хранения.
- **Hardware Acceleration**: Сжатие изображений и видео через нативные API браузера (OffscreenCanvas, WebCodecs).
- **Background Processing**: Все тяжелые операции выполняются в Web Worker.

---

## 🛠 Технологический стек
- **Dexie.js**: База данных для IndexedDB с поддержкой индексов и сложных запросов.
- **mp4-muxer**: Легковесная библиотека (8КБ) для упаковки WebCodecs-потока в MP4.
- **WebCodecs API**: Нативное аппаратное сжатие видео.
- **OffscreenCanvas**: Нативное сжатие изображений.
- **AES-GCM**: Сквозное шифрование ключом комнаты.

---

## 📋 Этапы реализации

### ЭТАП 1: Инфраструктура и Хранилище
1.  **Инсталляция зависимостей**: `npm install dexie mp4-muxer`.
2.  **Слой БД (`media-db.ts`)**: 
    - Определение схемы с полями `url`, `blob`, `pinnedCount`, `lastAccessedAt`.
    - Реализация логики "умной очистки": удаление только тех файлов, которые не находятся в "Избранном" (`pinnedCount == 0`).
3.  **Миграция**: Очистка старого хранилища `idb-keyval`.

### ЭТАП 2: Фоновая обработка (Web Worker)
1.  **Создание `media-processor.worker.ts`**:
    - Инкапсуляция логики сжатия изображений (OffscreenCanvas).
    - Инкапсуляция логики сжатия видео (WebCodecs).
    - Инкапсуляция шифрования (encryptBlob) для минимизации обмена данными между потоками.
2.  **Проверка поддержки**: Реализация детектора возможностей (WebCodecs support). Если не поддерживается — блокировка отправки.

### ЭТАП 3: Сервисный слой
1.  **`AttachmentService`**:
    - Валидация лимитов (10MB Photo / 30MB Video / 50MB Docs).
    - Генерация зашифрованных превью (Thumbnails) для мгновенной отрисовки.
    - Маскирование метаданных: оригинальное имя шифруется и сохраняется в JSON-поле `metadata`.
    - Использование флага `is_test` для идентификации тестового контента.

### ЭТАП 4: Интеграция в UI и Hooks
1.  **`useSendMessage`**: 
    - Перевод на использование `AttachmentService`.
    - Добавление индикаторов прогресса (сжатие, шифрование, загрузка).
2.  **Favorites Logic**: 
    - Обновление хуков избранного для управления `pinnedCount` в Dexie.

### ЭТАП 5: Backend TTL
1.  **`media_cleanup.pb.js`**: 
    - Расписание (Cron): раз в 24 часа.
    - Удаление записей коллекции `media` старше 30 дней.
    - Физическое удаление файлов с диска сервера.

## 🔐 Безопасность и Правила проекта
- **Positional Args**: Все функции принимают объекты параметров (`{ file, key, ... }`).
- **JSDoc**: Полное документирование на русском языке.
- **Zero Knowledge**: Шифрование происходит на клиенте **ДО** того, как файл попадет в сеть.
- **Mobile First**: Лимиты подобраны так, чтобы не перегружать память мобильных устройств.

---

## 📈 План верификации
1. **Developer Mode**: Вывод детальной статистики (Original Size vs Encrypted Size) в консоль.
2. **Unit-тесты**: Проверка корректности сжатия и шифрования через Vitest.
3. **Manual**: Отправка видео и фото с мобильного устройства, проверка очистки закешированных данных.
