# Правила проекта Knock-Knock

## � Общее видение
- **Аналог WhatsApp**: Мы реализуем защищенный мессенджер, ориентируясь на UX/UI и архитектурные решения WhatsApp. Это основной ориентир при принятии решений по кодовой базе и интерфейсу.

## �🎨 UI/UX и Стилизация

### Библиотеки
- **UI-основа**: [Radix UI](https://www.radix-ui.com) — используется **только внутри** `components/ui/`.
- **Иконки**: [Lucide React](https://lucide.dev/).
- **Стилизация**: Vanilla CSS с использованием **CSS Modules** (`*.module.css`).

### Принципы
1. **Кастомная дизайн-система поверх Radix**: В `features/` и `pages/` импортируем UI-компоненты **только** из `@/components/ui`. Radix-компоненты (`Text`, `Button`, `Avatar` и т.д.) используются **только внутри** `components/ui/` как основа для наших обёрток.
2. **Кастомезированные лэйауты**: `Flex`, `Box`, `Grid`, `Container`  — импортируются из `@/components/layout` как основа.
3. **Запрет Radix стилизующих пропсов**: В `features/` и `pages/` запрещены пропсы `color`, `size`, `radius`, `weight` на Radix-компонентах. Стилизация — только через `className` и CSS-переменные.
4. **Mobile First**: Верстка адаптивная, в первую очередь ориентирована на мобильные устройства.
5. **CSS Modules**: Все стили — в CSS-модулях рядом с компонентом. Никаких inline-стилей (`style={{}}`).
6. **Изоляция**: Каждый компонент в своей папке с `index.tsx` и `*.module.css`.
7. **UI Компоненты для диалогов**: Запрещены `window.confirm`, `window.alert`, `window.prompt`.
8. **Премиальный дизайн**: Градиенты, glassmorphism, микро-анимации.
9. **Единая палитра истины**: Все цвета — CSS-переменные в `app/src/index.css`. Запрещены hex-коды и `rgb()` внутри компонентов и CSS-модулей.
10. **Темизация**: Используй константы `THEME` из `@/lib/constants/theme`. Запрещены магические строки `"light"`, `"dark"`.

## 🛠 Технологический стек
- **Frontend**: React (последняя версия), Vite.
- **State Management**: TanStack Query (фетчинг и кеширование), Zustand (глобальный стейт).
- **Routing**: TanStack Router.
- **Backend/DB**: PocketBase v0.23+. Выбран за легковесность, встроенный Realtime (SSE) и удобную систему Hooks.
- **Криптография**: Web Crypto API (E2EE, P2P).

## ⚖️ Разделение ответственности: Компоненты vs Бизнес-логика

Четкое разделение UI и логики — критически важное требование для поддерживаемости проекта.

### 1. Компоненты (UI)
**Зона ответственности**: "Как показать?"
- **Отображение**: Как элементы выглядят и располагаются.
- **Взаимодействие**: Как элементы реагируют на клики, ввод и другие события (вызов колбэков).
- **Состояния UI**: Загрузка, ошибка, пустое состояние, открытый диалог.
- **НЕТ**: Принятия решений о данных, валидации бизнес-правил, расчетов денег/скидок и т.д.

### 2. Бизнес-логика
**Зона ответственности**: "Что показать и почему?"
- **Правила**: Определение доступности функций, расчеты, валидация.
- **Решения**: "Если пользователь VIP — дать скидку", "Если баланс < 0 — заблокировать".
- **Место**: Hooks (`useCalculator`, `usePermissions`), Services, Utility functions.

### 🔫 "Лакмусовая бумажка" (Тест на качество)
Задай вопрос при ревью компонента:
> "Если завтра мы заменим UI (React на Vue/Svelte/CLI), этот код должен остаться?"
- **Да** — это логика. Выноси в хук или сервис.
- **Нет** — это компонент. Оставляй в `tsx`.

### 🚩 Красные флаги (Red Flags)
- `useEffect` с сложными условиями и ветвлениями внутри UI компонента.
- Функции типа `calculatePrice`, `canUserEdit`, `resolveStatus` определены внутри компонента.
- Компонент делает fetch-запросы напрямую (а не вызывает хук).
- Компонент "знает" слишком много о структуре данных (маппинг API прямо в JSX).

## 📝 Код и Документация
- **Язык**: Весь код и комментарии пишутся на **русском языке**.
- **Типизация**: Строгий TypeScript. Избегаем `any` и кастования типов `as`.
- **JSDoc**: Документируй экспортируемые функции, компоненты и сложные участки логики.
- **Clean Code**: Следуй принципам SOLID и чистого кода. Выделяй логику в хуки и сервисы.
- **Single Source of Truth**: Типы данных, которые валидируются (DTO, Models), должны выводиться из **Zod Schemas** (`z.infer<typeof schema>`), расположенных в `src/lib/schemas`. Не дублируй интерфейсы вручную.

## 🧪 Качество и Стабильность
- **Линтинг и Тесты**: Все вносимые изменения должны проходить проверку линтером (`npm run lint`) и тесты на работоспособность/типизацию (`tsc`). Задача не считается выполненной, если есть ошибки.
- **Автономность**: Не нужно спрашивать разрешение на применение линтинга и проверки типов — применяй их самостоятельно и исправляй ошибки до завершения задачи.
- **Протокол помощи**: Если исправление ошибки не удается в течение **5 попыток**, необходимо прекратить самостоятельные действия, запросить помощь у USER и подробно объяснить причины неудач.

## ⛔ Архитектурные Антипаттерны (Что ломает проект)

### 1. ❌ Один стор для всего
**Проблема**: Глобальный стейт (Zustand/Redux), в который сваливают всё: UI-флаги, данные API, временные формы.
**Почему плохо**: Лишние ререндеры, неясные зависимости, сложности с масштабированием.
**Решение**:
- **Server State**: Используй TanStack Query для данных с бэкенда.
- **Form State**: Локальный стейт или специализированные библиотеки форм.
- **UI State**: Локальный `useState` или атомарные сторы (Zustand slices) только для того, что действительно нужно шарить.

### 2. ❌ Бизнес-логика в компонентах
**Проблема**: API-вызовы, маппинг данных, валидация и сложная логика прямо в JSX компонента.
**Почему плохо**: Компонент превращается в "God Component", сложный для тестирования и поддержки.
**Решение**:
- Выноси логику в **кастомные хуки** (`useChatData`, `useAuthLogic`).
- Используй **сервисные функции** для чистой логики (маппинг, валидация) вне React-контекста.

### 3. ❌ Утилиты на все случаи жизни (`utils.ts`)
**Проблема**: Один гигантский файл/папка `utils` со всем подряд.
**Почему плохо**: Свалка, где теряется контекст. Изменение одной утилиты может сломать непредсказуемые части системы.
**Решение**:
- Разбивай утилиты по доменам (`scrypto/`, `formatters/`, `validators/`).
- Храни утилиты рядом с кодом, который их использует, если они не общие.

### 4. ❌ Жёсткая привязка к API
**Проблема**: Использование структуры API напрямую в компонентах (`data.payload.items[0]...`).
**Почему плохо**: Изменение API ломает весь фронтенд.
**Решение**:
- Используй **DTO / Mappers**: Преобразуй данные с сервера в удобный для фронтенда формат сразу после получения (в `select` функции TanStack Query или сервисе).

### 5. ❌ Context Hell
**Проблема**: Использование React Context для всего подряд как замена архитектуре.
**Почему плохо**: Сложные зависимости, лишние ререндеры, трудность отслеживания потока данных.
**Решение**:
- Используй Context только для **глобальных конфигураций** (Theme, Auth, i18n).
- Для остального есть Composition, Props, или State Managers.

### 6. ❌ Оптимизация без проблемы
**Проблема**: Оборачивание всего в `useMemo`, `useCallback` "на всякий случай".
**Почему плохо**: Ухудшает читаемость и (иногда) производительность без реальной выгоды.
**Решение**:
- Оптимизируй только **после профилирования**, когда есть доказанная проблема.

### 7. ❌ Отсутствие границ (Leaky Abstractions)
**Проблема**: Компоненты знают слишком много о глобальном устройстве приложения.
**Почему плохо**: Невозможно переиспользовать компонент, сложно рефакторить.
**Решение**:
- Принимай только необходимые данные через пропсы.
- Используй **Container/Presentational** паттерн или слоты (Children) для композиции.

## 🤖 Процесс работы (Agentic Mode)
- **Explain -> Approve -> Do**: Всегда составляй план, предлагай 3 варианта решения с аргументацией, и приступай к реализации только после одобрения. Общение с пользователем на русском языке.
- **Аудит**: После выполнения всегда проводи запуск линтера и проверку типов.

## 🚀 Modern React & TanStack Best Practices
Эти правила обязательны для нового кода:

### 1. Избегаем `useEffect`
- **Проблема**: `useEffect` часто используется для синхронизации стейта, что ведет к лишним ререндерам и багам (race conditions).
- **Решение**:
    - Для данных: используй `useQuery` / `useSuspenseQuery`.
    - Для вычислений: используй derived state (просто переменные в теле функции) или `useMemo`.
    - Для событий: используй Event Handlers (`onClick`, `onSuccess` в мутациях).

### 2. TanStack Query Power
- Не используй ручной `isLoading` стейт, если есть `query.isPending`.
- Используй `useMutation` для любых изменений данных (создание, удаление, отправка).
- Инвалидируй кеш (`invalidateQueries`) вместо ручного обновления стейта списков.

### 3. Комментарии
- **Язык**: Русский.
- **Стиль**: Объясняй "ПОЧЕМУ", а не "ЧТО". Сложные хуки должны иметь JSDoc описание.

### 4. Декомпозиция (Decomposition)
- **Single Responsibility**: Одна функция/компонент — одна задача.
- **Размер**: Если компонент больше 200 строк — вероятно, пора делить. Если функция больше 50 строк — декомпозируй.
- **Хелперы**: Выноси утилитарную логику (`formatDate`, `encryptPayload`) в отдельные файлы или `utils.ts` внутри фичи.
