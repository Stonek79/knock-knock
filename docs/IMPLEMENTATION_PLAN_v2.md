## 🏛 План А: Качество и Полное Тестирование (QA Priority)
Цель: Обеспечить 100% уверенность в стабильности всех реализованных компонентов и фич на реальной базе (Staging).

### ЭТАП 1: Инфраструктура тестов (Staging Integration)
1. **Playwright Config**: 
   - Настройка `playwright.config.ts` для работы с `https://staging-api.knok-knok.ru:8445`.
   - Настройка автоматического переключения секретов (Anon Key, Service Key) для тестового ранера.
2. **Integration Setup**: 
   - Создание `vitest.integration.config.ts` для запуска тестов сервисов (`RoomService`, `MessageService`) против реальной БД Staging.
   - Реализация хелперов `db-cleanup` для очистки тестовых таблиц между запусками.

### ЭТАП 2: Покрытие E2E тестами (Все фичи)
Необходимо написать сценарии для следующих критических путей:
1. **Auth Flow**: 
   - Регистрация и вход (Email/Password).
   - Проверка сессии после перезагрузки страницы.
2. **Messaging Flow**:
   - Отправка и получение текстовых сообщений (Real-time).
   - Удаление и редактирование сообщений (Sync).
   - Индикация "Печатает..." и "В сети".
3. **E2E Crypto Flow**:
   - Проверка генерации ключей при первом входе.
   - Успешная расшифровка контента на другом клиенте.
   - Проверка "нечитаемости" зашифрованных данных в БД напрямую.
4. **Media Flow**:
   - Отправка изображений (E2E шифрование + Storage).
   - Запись и воспроизведение голосовых сообщений.
5. **Chat Features**:
   - Создание личного чата (DM).
   - Создание группового чата и добавление участников.
   - Работа с "Избранным" (Saved Messages).
6. **Ghost Mode**:
   - Активация режима через ПИН-код.
   - Проверка скрытия чатов при входе через Decoy PIN.

### ЭТАП 3: UI-тестирование (Storybook & Biome)
1. **Storybook Coverage**: 
   - Создание stories для всех 29 UI-компонентов (Button, Alert, TextField, Avatar и т.д.).
   - Настройка Accessibility (A11y) тестов внутри Storybook.
2. **Biome Linting**: 
   - Полный аудит кодовой базы и автоматическое исправление стилистических ошибок.

### ЭТАП 4: Аудит документации и JSDoc (Knowledge Management)
1. **JSDoc & Comments Audit**:
   - Проверка всех экспортируемых функций, хуков и компонентов на наличие JSDoc-описаний (аргументы, возвращаемые типы, цель).
   - Проверка наличия пояснительных комментариев ("ПОЧЕМУ", а не "ЧТО") в сложных алгоритмах шифрования и хендлерах событий.
   - Все комментарии должны быть на **русском языке** согласно правилам проекта.
2. **Project Documentation Support**:
   - Проверка актуальности файлов в `/docs` (Roadmap, Security, Architecture).
   - Создание/обновление README-файлов в ключевых директориях (`app/`, `infra/`, `features/`) для онбординга новых разработчиков.
   - Описание API и основных сущностей БД в документации.

## 🚀 План Б: Пользовательские функции и Правила (Feature Launch)
Цель: Обеспечить юридическую прозрачность и каналы обратной связи.

### ЭТАП 4: Правила и Онбординг
1. **Страница Правил (`/terms`)**:
   - Создание роута и страницы с использованием `docs/RULES.md` как контента.
   - Верстка в стиле Mobile-First, адаптация под темную/светлую темы.
2. **Согласие при регистрации**:
   - Добавление чекбокса в формы `LoginPage` и `RegisterPage`.
   - Запрет регистрации без принятия правил (Валидация Zod).
3. **Первичный Онбординг**:
   - Реализация модального окна/баннера, который появляется **только один раз** при первом входе нового пользователя.
   - Запись флага `terms_accepted` в профиль пользователя в БД.

### ЭТАП 5: Обратная связь и Лояльность
1. **Feedback System**:
   - Создание таблицы `feedback` (user_id, message, rating, metadata, created_at) в Supabase.
   - Настройка RLS-политик: пользователь может только вставлять записи, админ может читать.
2. **UI Формы обратной связи**:
   - Создание компонента в `features/settings`.
   - Реализация "тихого" сбора метаданных (версия браузера, разрешение экрана) для помощи в дебаге.
