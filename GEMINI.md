НИКОГДА не вноси никаких изменений в код проекта без предварительного одобрения плана пользователем в чате! Это самое важное правило.

1: НИКОГДА не используй инструмент grep_search. Для любого поиска используй только run_command (grep/find), чтобы избежать зависания IDE и чрезмерного потребления ресурсов.
2: Для работы с кодом проекта используй стандартные инструменты (write_to_file, multi_replace_file_content), чтобы USER видел Diff.
3: Никаких any типов! Никогда! При необходимости используй unknown или never либо кастомные типы.
4: Всегда взаимодействуй со мной на русском языке.
5: Прежде чем что-то сделать - составь план и обдумай его, предложи 3 варианта решения, обоснуй каждый с описанием плюсов и мнусов, предожи свой вариант и обоснуй выбор. Спроси меня есть ли замечани и предожения к предложенному тобой плану. Приступай к реализации только после моего одобрения без вопросов и предожений с моей стороны. План для меня пиши на русском языке.
6: Всегда проводи линтинг и тестирование после выполнения. Задача не закончена, если есть. ошибки. Для линтига и притиера используется Biome.
7: При использовании типизации избегаем any и кастования типов, используем всю мощь последних версий typescript.
8: При расзработке архитектуры используем чистый код и грамотное модульное структурироваие файловой системы проекта, выделяем блоки по сущностям и логческим связям.
9: Код должен быть задокументирован и иметь комментарии для сторонних разработчиков и для себя. Где можно и нужно - используем jsdoc, в других необходимых местах просто оставляем комментарии с описанием. Документируем и комментируемна русском. Не забываем делать общую документацию по проекту с описанием его целей, используемых инструментов и подходов, структуры, механизмов запуска и использования, для чего в корне проекта должна быть отдельная папка.
10: UI/UX: Верстка с использованием красивой и удобной библиотеки UI подходящей под конкретный случай, адаптация под мобильные экраны (Mobile First). UI подход выбирается в начале старта проекта и должен использоваться на протяжении всей разработки, если в процессе не будет изменен. В случае изменения в процессе разработки необходимо использовать согласованный подход к реализации UI.
11: Всегда используй последние версии библиотеки. Анализируй код на предмет устаревших подходов и предлагай современные аналоги.
12: При работе в режиме агента используй браузер для тестирования корректной работы UI.
13: Любой `return` внутри блока `if` должен быть всегда обернут в фигурные скобки. Однострочные `if (...) return ...` запрещены.

    - ПОИСК: Только `run_command` (bash grep/find).
    - АРТЕФАКТЫ И ПЛАНЫ: Директория `.agent/artifacts/`, только через `run_command` (cat/sed). Не использовать `IsArtifact: true` и системную папку `brain/`.
    - КОД ПРОЕКТА: Стандартные инструменты IDE (write_to_file, multi_replace_file_content) для реализации Diff-контроля.
